# 📐 Введение: Пирамида тестирования

## Зачем вообще тестировать?

Представь, что ты написал код для онлайн-магазина книг. Как узнать, что он работает правильно?

**Вариант 1:** Запускаешь сайт, кликаешь руками, проверяешь.  
**Проблема:** Делать это после каждого изменения — долго. Можно что-то забыть проверить. При командной работе — умножь на всех разработчиков.

**Вариант 2:** Пишешь автоматические тесты.  
**Результат:** Нажал кнопку → 5 секунд → узнал, что всё работает (или нет, и где именно).

---

## Что такое пирамида тестирования?

Пирамида тестирования — это модель, описывающая **сколько тестов** и **какого типа** стоит писать.

```
            ╔═══════════╗
            ║   E2E     ║  ← 5-10 тестов
            ╚═══════════╝
         ╔═════════════════╗
         ║  Интеграционные ║  ← 20-30 тестов
         ╚═════════════════╝
      ╔═════════════════════════╗
      ║     Компонентные        ║  ← 50-100 тестов
      ╚═════════════════════════╝
   ╔═════════════════════════════════╗
   ║          Unit-тесты             ║  ← 200-500 тестов
   ╚═════════════════════════════════╝
```

**Правило пирамиды:**
- Основание (юниты) → **много, быстро, дёшево**
- Вершина (E2E) → **мало, медленно, дорого**

---

## Почему такая форма?

| Характеристика | Unit | Component | Integration | E2E |
|---------------|------|-----------|-------------|-----|
| ⚡ Скорость | мс | сек | сек-мин | мин |
| 💰 Стоимость поддержки | низкая | средняя | высокая | очень высокая |
| 🎯 Точность локализации бага | высокая | средняя | низкая | очень низкая |
| 🔄 Стабильность | высокая | высокая | средняя | низкая |
| 🌐 Покрытие реального сценария | низкое | среднее | высокое | максимальное |

## Антипаттерны пирамиды

### 🍦 «Мороженое» (Ice cream cone)
```
   ╔══════════════════════════╗
   ║         E2E (много!)     ║  ← написали только ручные/E2E
   ╚══════════════════════════╝
      ╔══════════════════╗
      ║   Интеграционные ║
      ╚══════════════════╝
         ╔══════════╗
         ║   Unit   ║  ← совсем мало
         ╚══════════╝
```
**Проблема:** Тесты медленные, нестабильные. CI-pipeline занимает часы.

Больший акцент на интеграционный слой Testing Library.  
Не догма — выбирайте под задачу.

## Приложение, которое мы тестируем

**BookStore** — онлайн-магазин книг.

```
[React Frontend]  →  [Spring Boot API]  →  [PostgreSQL]
                           ↓
              [Google Books API (ISBN lookup)]
              [Notification Service (email)]
```

**Что умеет:**
- CRUD записей о книгах
- Поиск по названию и ISBN
- Добавление в корзину / оформление заказа
- Уведомления по email
- Обогащение данных из внешнего ISBN-сервиса
